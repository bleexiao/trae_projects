# Practice06 修复总结

## 修复的问题

### 1. URL 中文编码问题
**问题**: 当 URL 包含中文字符时（如 `https://wttr.in/成都`），HTTP 请求失败，报错 `'ascii' codec can't encode characters`。

**解决方案**:
- 在 `fetch_webpage` 函数中添加 URL 编码处理
- 使用 `urllib.parse.quote` 对路径部分进行 UTF-8 编码
- 移除 URL 两端的反引号和空格

**修复位置**: `chat_client.py` 第 273-313 行

### 2. 参数值清理问题
**问题**: LLM 返回的工具参数中可能包含反引号和多余空格（如 `{'url': ' `https://wttr.in/成都` '}`），导致 URL 解析错误。

**解决方案**:
- 在解析工具调用参数后，清理参数值中的反引号和多余空格
- 对所有字符串类型的参数值执行 `.strip().strip('`')`

**修复位置**: `chat_client.py` 第 616-623 行和第 673-680 行

### 3. JSON 解析容错性
**问题**: LLM 返回的 JSON 可能格式不完整或包含额外内容，导致解析失败。

**解决方案**:
- 增强 `parse_json_response` 函数的容错能力
- 尝试移除末尾的逗号
- 尝试提取第一个完整的 JSON 对象
- 增加调试输出，显示解析失败时的原始文本

**修复位置**: `chat_client.py` 第 508-540 行

### 4. 消息历史累积问题
**问题**: 每次迭代都重新初始化消息历史，导致 LLM 无法看到之前的工具执行结果。

**解决方案**:
- 在循环外部初始化消息历史
- 每次迭代只更新系统提示词，保留之前的对话历史
- 在工具执行后将结果添加到消息历史

**修复位置**: `chat_client.py` 第 600-620 行

### 5. 缺失的 add_call 调用
**问题**: 在 JSON content 格式的工具调用中，缺少 `context.add_call()` 调用，导致调用历史为空。

**解决方案**:
- 在工具执行后添加 `context.add_call(function_name, arguments, result)`
- 确保两种格式的工具调用（tool_calls 和 JSON content）都记录到上下文

**修复位置**: `chat_client.py` 第 740 行
